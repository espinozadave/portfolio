Linux permissions management

Overview

This project demonstrates the process of reviewing and securing Linux file and directory permissions to enforce proper authorization. I analyzed permission settings, identified misconfigured access, and applied corrective actions to restrict unauthorized users and protect sensitive data.

Scope

  • Examined file and directory permissions using ls -la
  • Interpreted the 10-character Linux permission string
  • Identified files with unauthorized write access
  • Modified permissions using chmod to enforce least privilege
  • Secured hidden archived files from modification
  • Restricted directory access to authorized users only
  • Verified all changes by rechecking permission settings

Project Descrption

As a security professional supporting a research team, I reviewed and managed Linux file system permissions to ensure users were granted appropriate access. I examined existing permissions, identified mismatches with required authorization, and modified them to remove unauthorized access. Using commands such as ls -la and chmod, I verified and enforced proper access controls to help maintain system security and protect sensitive research data.

Scenario

This scenario is based on a fictional company:

You are a security professional at a large organization. You mainly work with their research team. Part of your job is to ensure users on this team are authorized with the appropriate permissions. This helps keep the system secure.

Your task is to examine existing permissions on the file system. You’ll need to determine if the permissions match the authorization that should be given. If they do not match, you’ll need to modify the permissions to authorize the appropriate users and remove any unauthorized access.

File & Directory Inspection

I first verified that the projects directory existed by using the ls command to list available directories. After confirming its presence, I navigated to the directory using cd projects.

Within the directory, I ran ls -la to display detailed information for all files and subdirectories, including hidden items. This command provided the permission settings for each file and directory, represented by a 10-character permission string, which identifies the access levels for the user, group, and others. Understanding Linux Permission String

Linux file permissions are displayed as a 10-character string, such as drwxrwxrwx, which defines the access level for different users.

  • First character -- Identifies the file type. A d represents a directory, while a - indicates a regular file. Other symbols may appear for special file types, such as links.
  • Character 2-4 -- Represent the permissions assigned to the owner (user). These include read (r), write (w), and execute (x). A hyphen (-) in any position means that specific permission is not granted.
  • Characters 5–7 -- Define the permissions for the group. These follow the same read, write, and execute structure and determine what members of the file’s group are allowed to do.
  • Characters 8–10 -- Specify permissions for others, which include all users who are not the owner and not part of the group. These permissions control general access to the file or directory.

In addition to the permission string, the expanded directory listing also displays ownership details. The second column shows the user who owns the file, while the third column identifies the group that owns it. Understanding both permissions and ownership is important for properly managing access control in Linux systems.

Restricting File Permissions

To strengthen system security, I reviewed file permissions to identify any unauthorized access. Specifically, I checked whether any files in the projects directory allowed others to have write permissions.

Using the ls -la command, I examined the permission settings and discovered that project_k.txt granted write access to others, which posed a security risk. To remove this unauthorized permission, I executed: chmod o-w project_k.txt

This command removed the write permission for others while keeping the existing permissions for the user and group unchanged. I then ran ls -la again to verify that the changes were successfully applied and that write access for others had been removed. The screenshot above shows the result of the command executed.




The file project_m.txt is considered a restricted file and should only be accessible by the user. The group and others should not have read or write permissions.

I reviewed the current permissions using ls -la and observed that the group had read access to project_m.txt, which did not align with the required authorization. To remove this incorrect permission, I executed: chmod g-r project_m.txt

This command removed the read permission for the group. I then ran ls -la again to verify that the permission change was successfully applied and that only the user retained access to the file. The screenshot above shows the result of the command executed.

Securing Hidden Files

To further reduce unauthorized access, I reviewed the permissions of hidden files within the directory. The file .project_x.txt is an archived hidden file and should not be modified. However, both the user and group should retain read access.

Using ls -la, I observed that both the user and group had write permissions, which did not align with the required access controls. To correct this, I executed: chmod u-w,g-w,g+r .project_x.txt

This command removed write permissions for both the user and group while ensuring read access remained intact. I then ran ls -la again to verify that the changes were successfully applied and that the file permissions reflected the intended restrictions. Restricting Directory Access

In this task, I reviewed the permissions of the drafts subdirectory within /home/researcher2/projects. Only the researcher2 user should have access to this directory, meaning that only the user should retain execute privileges.

After checking the current permissions using ls -la, I observed that the group had execute access to the drafts directory, which was not authorized. To restrict this access, I executed: chmod g-x drafts/

This command removed the execute permission from the group. I then ran ls -la again to verify that the change was successfully applied and that only the authorized user retained access to the directory. The screenshot above shows the result of the command executed.

Summary

I reviewed and corrected Linux file and directory permissions to ensure proper authorization and prevent unauthorized access. Using ls -la, I examined existing permissions, including hidden files, and interpreted the 10-character permission string to identify misconfigured access levels.

I applied chmod to remove unnecessary write and execute permissions, restrict access to sensitive files, secure hidden archived content, and ensure only authorized users could access critical directories. Each modification was validated by rechecking permissions to confirm the changes.

These actions strengthened access control, reduced security risks, and ensured that files and directories followed the principle of least privilege.
