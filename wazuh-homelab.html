🔐 Home Lab Project: SIEM Monitoring & Brute-Force Attack Detection

Overview

This project demonstrates a simulated security operations environment using virtual machines to emulate an attacker, a monitored endpoint, and a centralized SIEM server. The objective is to perform a brute-force attack from an attacker machine and observe how security logs are generated, forwarded, and analyzed by the SIEM platform.

The lab consists of three systems communicating within an isolated internal network:

  • Kali Linux – Attacker (192.168.10.8)
  • Ubuntu – Wazuh Server (192.168.10.3)
  • Windows – Wazuh Agent (192.168.10.24)

All machines are configured within the same internal network to safely simulate cyber attack scenarios without external exposure.

Objective

  • Simulate brute-force attack attempts from attacker machine
  • Monitor security events generated on Windows endpoint
  • Forward logs from Windows agent to centralized SIEM server
  • Analyze alerts and detection visibility in the Wazuh dashboard

Tools & Technologies Used

  • Wazuh – SIEM and host-based intrusion detection
  • Ubuntu - Wazuh Manager
  • Windows 10 – Monitor Endpoint
  • Kali Linux – Attacker Machine
  • Oracle VM VirtualBox – Virtualized lab environment

Brute-force preparation and execution from Kali attacker. A custom password list was created and used with Hydra to perform an RDP brute-force attempt against Windows-agent (192.168.10.24).

Successfully triggered a False Positive during an RDP brute-force simulation using Hydra. Although the password 'Password123' was incorrect, the service returned a 'Not Authorized' flag rather than 'Access Denied.' This occurred because the target user 'homel' had not yet been added to the Remote Desktop Users group, causing Hydra to misidentify the permission error as a successful credential match.

Logs Generated on Windows

Security events observed on the Wazuh manager (Ubuntu) confirming log generation from the Windows-agent following a simulated brute-force attack.

This log represents a telemetry capture of a failed authentication attempt. It shows that at 10:08 PM, a workstation named 'kali' (IP 192.168.10.8) attempted to log into the 'homel' account on the Windows VM using the NTLM protocol. The Windows Security Auditing system intercepted the bad password, generated Event ID 4625, and the Wazuh agent immediately forwarded this data to the manager for analysis.

Findings (SIEM Home Lab — Brute-Force Attempt)

Incident Overview:

  • Date: April 8, 2026
  • Time: 10:08 PM
  • Target: Windows VM (Wazuh Agent)
  • Attacker: Kali VM
  • Wazuh Manager: Ubuntu VM
Event Summary:
  • A brute-force login attempt was simulated from Kali (192.168.10.8) targeting the Windows machine (192.168.10.24).
  • The authentication attempts failed and did not result in unauthorized access.
  • The Windows endpoint generated security logs for each failed login attempt.
  • The Wazuh agent successfully forwarded these logs to the centralized Wazuh manager.
  • The Wazuh dashboard displayed alert details including source IP address, hostname, and event severity.
  • The SIEM correlated the failed login events and provided visibility into the attempted intrusion.

Lessons Learned

Importance of Agent Configuration:

  • Proper configuration of the Wazuh agent on the Windows host ensured reliable log forwarding to the SIEM manager.
Network Segmentation:
  • Conducting the test within an isolated internal network allowed safe simulation of attack activity without external exposure.
Detection Over Attack Success:
  • Even though the brute-force attack failed, the SIEM successfully detected suspicious authentication attempts, demonstrating the value of proactive monitoring.
Visibility and Incident Response Practice:
  • Analyzing failed login events, source IP addresses, and hostnames provided hands-on experience in identifying and investigating security incidents.
Centralized Log Monitoring:
  • The lab highlighted the importance of centralized logging in improving detection and simplifying incident analysis across endpoints.

Recommendations

  • Implement account lockout policies to mitigate brute-force attacks.
  • Configure alert thresholds for repeated failed login attempts.
  • Enable additional log sources for broader visibility.
  • Integrate automated response actions for high-severity alerts.
